# Library Research: Rust NIFs in Elixir via Rustler

## Recommended

### rustler
- **Hex**: https://hex.pm/packages/rustler
- **Docs**: https://hexdocs.pm/rustler
- **Downloads**: 4,553,206 all-time; 37,153 last 30 days
- **Last release**: 0.37.3 on 2026-02-11
- **Why**: The canonical library for writing Erlang NIFs in safe Rust. Rust panics are caught before unwinding into C, so the BEAM VM cannot be crashed. Provides derive macros for automatic Elixir/Rust term conversion, resource objects with automatic memory management, and a Mix compiler integration that drives `cargo build` transparently.

## 2. mix.exs Configuration

```elixir
# mix.exs
def deps do
  [
    {:rustler, "~> 0.37.3", runtime: false}
  ]
end
```

The `runtime: false` flag is required — rustler is a compile-time Mix compiler helper only.

After adding the dep:

```bash
mix deps.get
mix deps.compile
mix rustler.new   # interactive scaffold generator
```

The `mix rustler.new` generator prompts for:
1. Elixir module name (e.g. `MyApp.NativeModule`)
2. Rust crate name (e.g. `my_native`)

It creates `native/<crate>/` with a working skeleton.

## 3. Elixir Module Setup

```elixir
defmodule MyApp.NativeModule do
  use Rustler,
    otp_app: :my_app,
    crate: :my_native   # matches [lib] name in Cargo.toml; omit if same as otp_app

  # Stub clauses are REQUIRED for every exported NIF.
  # They run only if the NIF fails to load.
  def add(_a, _b), do: :erlang.nif_error(:nif_not_loaded)
  def my_function(_arg), do: :erlang.nif_error(:nif_not_loaded)
end
```

Alternatively, configure via `config/config.exs` (useful for overriding per-env):

```elixir
# config/config.exs
config :my_app, MyApp.NativeModule,
  crate: :my_native,
  mode: :release,        # :debug | :release (default: :release)
  load_data: 0           # term passed to the NIF on load
```

### Full Option Reference for `use Rustler`

| Option | Default | Description |
|---|---|---|
| `:otp_app` | required | Your application atom |
| `:crate` | `:otp_app` value | Rust crate name (must match `[lib] name` in Cargo.toml) |
| `:mode` | `:release` | `:debug` or `:release` |
| `:path` | `"native/<crate>"` | Override crate source location |
| `:load_from` | auto | Override artifact location: `{:app_name, "priv/native/<artifact>"}` |
| `:load_data` | `0` | Any Elixir term passed to NIF `on_load` callback |
| `:load_data_fun` | nil | `{Module, :function}` for dynamic load data; mutually exclusive with `:load_data` |
| `:features` | `[]` | Cargo features to enable |
| `:default_features` | `true` | Enable/disable crate default features |
| `:env` | `%{}` | Environment variables for the `cargo` invocation |
| `:cargo` | `:system` | How to invoke Rust compiler (`:system`, `:rustup`, channel string, or binary path) |
| `:target` | host | Rust platform target triple |
| `:skip_compilation?` | `false` | Skip Rust build; use with precompiled artifacts |

## 4. Directory Structure (native/ conventions)

Generated by `mix rustler.new`:

```
my_app/
  mix.exs
  config/
    config.exs
  lib/
    my_app/
      native_module.ex      # defmodule with `use Rustler` and stub clauses
  native/
    my_native/              # one directory per Rust crate; name matches Cargo.toml [lib] name
      Cargo.toml
      src/
        lib.rs              # entry point; contains #[rustler::nif] fns + rustler::init!()
  priv/
    native/                 # compiled .so/.dll/.dylib lands here at build time (auto-generated)
```

For projects with multiple NIFs, add one subdirectory under `native/` per crate.

### Cargo.toml minimum viable config

```toml
[package]
name = "my_native"
version = "0.1.0"
edition = "2021"

[lib]
# "cdylib" is REQUIRED — produces a C-compatible dynamic library for the BEAM to dlopen
crate-type = ["cdylib"]

[dependencies]
rustler = "0.37"
# For newer OTP NIF APIs:
# rustler = { version = "0.37", features = ["nif_version_2_16"] }  # OTP 24
# rustler = { version = "0.37", features = ["nif_version_2_17"] }  # OTP 26
# For serde serialization:
# rustler = { version = "0.37", features = ["serde"] }
```

## 5. Defining NIF Functions: Rust Side

### Basic scalar functions

```rust
// native/my_native/src/lib.rs

#[rustler::nif]
fn add(a: i64, b: i64) -> i64 {
    a + b
}

#[rustler::nif]
fn reverse_string(s: String) -> String {
    s.chars().rev().collect()
}

// rustler::init! registers the Elixir module and the exported functions.
// The module name must be the full Elixir atom: "Elixir.MyApp.NativeModule"
rustler::init!("Elixir.MyApp.NativeModule");
```

Functions listed in `rustler::init!` are automatically discovered via the `#[rustler::nif]` attribute — no explicit list required in 0.37.x.

### Returning Results (NifResult)

```rust
use rustler::NifResult;
use rustler::Error;

#[rustler::nif]
fn safe_divide(a: f64, b: f64) -> NifResult<f64> {
    if b == 0.0 {
        Err(Error::BadArg)
    } else {
        Ok(a / b)
    }
}
```

`NifResult<T>` maps to `{:ok, value}` / `{:error, reason}` on the Elixir side.

Available error variants in `rustler::Error`:
- `Error::BadArg` — raises `ArgumentError` in Elixir
- `Error::Atom("my_error")` — returns `{:error, :my_error}`
- `Error::RaiseTerm(term)` — raises with a custom term

### Atoms

```rust
use rustler::Atom;

// Define reusable atoms (avoids repeated allocation)
rustler::atoms! {
    ok,
    error,
    invalid_input,
}

#[rustler::nif]
fn status_atom() -> Atom {
    ok()
}
```

### Structs as Elixir structs (NifStruct)

```rust
use rustler::NifStruct;

// Maps to %MyApp.Point{x: ..., y: ...} in Elixir
#[derive(Debug, NifStruct)]
#[module = "MyApp.Point"]
pub struct Point {
    pub x: f64,
    pub y: f64,
}

#[rustler::nif]
fn translate(point: Point, dx: f64, dy: f64) -> Point {
    Point {
        x: point.x + dx,
        y: point.y + dy,
    }
}
```

### Maps (NifMap)

```rust
use rustler::NifMap;

// Encodes/decodes as a plain Elixir map %{lhs: ..., rhs: ...}
#[derive(NifMap)]
pub struct AddMap {
    pub lhs: i32,
    pub rhs: i32,
}
```

### Tuples (NifTuple)

```rust
use rustler::NifTuple;

// Encodes/decodes as an Elixir tuple {lhs, rhs}
#[derive(NifTuple)]
pub struct Pair {
    pub lhs: i32,
    pub rhs: i32,
}
```

### Records (NifRecord)

```rust
use rustler::NifRecord;

// Encodes/decodes as an Erlang record {:my_record, lhs, rhs}
#[derive(NifRecord)]
#[tag = "my_record"]
pub struct MyRecord {
    pub lhs: i32,
    pub rhs: i32,
}
```

### Binary data

```rust
use rustler::{Binary, OwnedBinary, NifResult};
use std::io::Write;

// Accept binary, return owned binary
#[rustler::nif]
fn double_bytes(input: Binary) -> NifResult<OwnedBinary> {
    let mut output = OwnedBinary::new(input.len() * 2).unwrap();
    output.as_mut_slice()[..input.len()].copy_from_slice(&input);
    output.as_mut_slice()[input.len()..].copy_from_slice(&input);
    Ok(output)
}
```

### Dirty schedulers (long-running work)

```rust
// Marks this NIF as CPU-bound dirty — runs on a separate thread pool,
// preventing it from blocking the BEAM scheduler.
#[rustler::nif(schedule = "DirtyCpu")]
fn heavy_computation(data: Vec<u8>) -> Vec<u8> {
    // ... long CPU work ...
    data
}

// For blocking I/O
#[rustler::nif(schedule = "DirtyIo")]
fn blocking_read(path: String) -> NifResult<Vec<u8>> {
    std::fs::read(&path).map_err(|_| rustler::Error::BadArg)
}
```

### Working with generic Terms and Env

```rust
use rustler::{Env, Term, NifResult};

// Access the Erlang environment directly when you need dynamic term construction
#[rustler::nif]
fn wrap_in_ok<'a>(env: Env<'a>, term: Term<'a>) -> Term<'a> {
    let ok = rustler::types::atom::ok();
    (ok, term).encode(env)
}
```

### Load callback (on_load)

```rust
fn on_load(env: Env, _load_info: Term) -> bool {
    // Register resource types, perform one-time setup
    rustler::resource!(MyResource, env);
    true
}

rustler::init!("Elixir.MyApp.NativeModule", load = on_load);
```

## 6. Corresponding Elixir Stubs

Every Rust NIF exported by `rustler::init!` must have a matching stub in the Elixir module:

```elixir
defmodule MyApp.NativeModule do
  use Rustler, otp_app: :my_app, crate: :my_native

  def add(_a, _b), do: :erlang.nif_error(:nif_not_loaded)
  def reverse_string(_s), do: :erlang.nif_error(:nif_not_loaded)
  def safe_divide(_a, _b), do: :erlang.nif_error(:nif_not_loaded)
  def double_bytes(_input), do: :erlang.nif_error(:nif_not_loaded)
  def heavy_computation(_data), do: :erlang.nif_error(:nif_not_loaded)
  def translate(_point, _dx, _dy), do: :erlang.nif_error(:nif_not_loaded)
end
```

The stub arity must match the Rust function arity exactly.

## No Library Needed

The standard Rust `std` library handles most data transformation needs. For pure Elixir computation that does not require native speed, plain Elixir/Erlang is sufficient. Rustler is only warranted when:
- CPU-bound work causes measurable BEAM scheduler contention
- You need to call existing C/C++ libraries via Rust `unsafe` FFI
- Cryptographic or numeric routines require SIMD/native performance

## Compatibility Notes

- Elixir version requirement: >= 1.15
- OTP version requirement: Three latest major OTP versions (OTP 24+ recommended for `nif_version_2_16` features)
- Rust MSRV: 1.91
- NIF version default: 2.15 (OTP 22+); opt into 2.16 (OTP 24+) or 2.17 (OTP 26+) via Cargo features
- Known conflicts: None with standard Phoenix/Ecto stack; `runtime: false` prevents any runtime dependency conflict
- The generated `.so`/`.dylib`/`.dll` is placed in `priv/native/` and must be included in releases (Distillery/Mix Release both handle `priv/` automatically)

## Considered but Rejected

### zigler
- **Why not**: Targets Zig instead of Rust. Smaller ecosystem, far fewer downloads, fewer derive macros for Elixir type interop. Only relevant if your native code is already in Zig.

### erlang_nif (raw NIF via C/`:nif`)
- **Why not**: Writing NIFs directly in C is unsafe — any crash kills the BEAM process. Rustler exists precisely to avoid this.
